# 最终开发规范

本文件为当前项目唯一主规范。

后续新增页面、接口、云函数、交互、测试要求，默认全部按本文件执行。

---

## 一、协作定位

协作模式默认如下：

- 你负责定义“什么才是好用的软件”
- 技术实现负责把需求落成稳定、可维护、可测试的代码
- 需求讨论必须同时考虑：
  - 用户体验
  - 逻辑正确性
  - 异常场景
  - 长期维护成本

沟通默认原则：

1. 禁止只讲思路不落代码
2. 禁止直接甩一大段无注释代码
3. 必须先解释逻辑，再落代码
4. 必须主动指出潜在逻辑漏洞、体验坑点和异常风险

---

## 二、默认优先级

所有开发默认按以下优先级决策：

1. 安全性
2. 性能
3. 用户体验

如三者冲突，优先保证：

- 数据安全
- 状态一致
- 用户不误操作

---

## 三、用户体验要求

1. 操作流程必须尽量极简
   - 核心功能点击次数默认不超过 3 次

2. 所有加载状态必须有明确反馈
   - 必须显示加载中
   - 禁止白屏
   - 禁止无响应

3. 所有报错必须说人话
   - 必须告诉用户发生了什么
   - 必须告诉用户下一步怎么做
   - 禁止直接暴露原始报错码给用户

4. 页面必须有稳定的空态
   - 无数据时不能白屏
   - 要有明确空态文案

---

## 四、异常场景防御

以下异常场景必须全部考虑并落代码：

### 1. 弱网 / 断网

- 必须有自动提示或失败提示
- 必须提供重试按钮
- 不能导致数据错乱
- 请求失败后必须有兜底文案

### 2. 并发冲突

- 必须防重复提交
- 必须考虑幂等
- 涉及共享资源必须加锁或事务
- 严禁超卖、重复扣减、状态错乱

### 3. 误操作

- 删除 / 取消 / 清空 / 关闭关键状态时必须二次确认
- 高风险操作必须可撤销或可回退时优先考虑撤销

### 4. 输入校验

- 必须校验空值
- 必须限制超长文本
- 必须过滤或拦截特殊字符 / 非法输入
- 必须处理异常格式

### 5. 云开发异常

- 数据库失败必须捕获
- 云函数超时必须捕获
- 必须返回用户友好提示
- 必须保留开发排查日志

### 6. 无权限 / 未登录

- 必须自动引导
- 页面不能崩溃
- 禁止直接抛原始错误给用户

---

## 五、代码强制约束

以下规则绝对禁止违反：

1. 必须写代码，不允许只写逻辑描述
2. 禁止甩大段无注释代码
3. 所有异步请求必须包 `try/catch`
4. 所有接口必须有加载提示
5. 所有错误必须给用户友好提示
6. 必须考虑测试视角：
   - 边界
   - 异常
   - 空值
   - 重复点击
7. 代码风格统一：
   - ES6+
   - `async/await`
   - 严格模式
8. 云开发必须正确配置 `envId`
   - 禁止硬写危险代码
9. 页面必须做：
   - `onLoad` 异常捕获
   - 无数据处理
   - 网络异常处理
10. 每次交付默认输出必须包含：

- 功能说明
- 代码
- 测试点
- 验收标准

---

## 六、注释规范

1. 关键逻辑必须有简短注释
2. 注释必须解释“为什么这样做”
3. 禁止无意义注释
4. 复杂状态流转、事务、幂等、防重复提交必须注释

---

## 七、前端页面最低要求

每个页面默认至少必须考虑：

1. `onLoad` 使用异常捕获
2. 加载中状态
3. 空状态展示
4. 网络异常提示
5. 重试入口
6. 关键操作防重复点击
7. 用户可读错误提示

如页面存在表单或提交动作，还必须考虑：

1. 输入合法性校验
2. 禁止重复点击
3. 失败可重试
4. 提交中态可视化

---

## 八、云函数最低要求

所有云函数默认必须满足：

1. 必须包含事务处理
2. 多文档更新必须放在事务中
3. 必须做参数校验
4. 必须做身份校验
5. 必须做权限校验
6. 必须考虑幂等
7. 必须记录排查日志
8. 必须返回统一成功/失败结构

建议默认结构：

1. 参数校验
2. 身份校验
3. 权限校验
4. 开启事务
5. 查询当前状态
6. 执行业务更新
7. 提交事务
8. 返回结果

以下关键逻辑不允许只写在前端：

- 会员等级累计
- 勋章解锁
- 场次码核销
- 库存扣减
- 多角色状态流转

---

## 九、技术选型原则

默认优先推荐：

- 维护成本最低
- 对单人开发最友好
- 适合快速迭代
- 稳定性优先于花哨架构

如果是微信小程序项目，默认优先考虑：

- 微信原生小程序
- CloudBase

前提是：

- 运维成本可控
- 权限和数据一致性能满足业务要求

---

## 十、测试视角要求

所有功能开发完成后，默认必须从以下角度回看：

1. 正常流程是否顺畅
2. 异常流程是否可恢复
3. 边界条件是否稳定
4. 空值是否安全
5. 重复点击是否安全
6. 弱网下是否能给明确反馈
7. 用户是否能看懂错误提示

---

## 十一、变更影响审查要求

每次修改任何问题前，默认必须先做关联代码审查，不能只盯单点修复。

强制要求：

1. 必须先检查该问题对应的数据流、调用链和相关页面/云函数/后台逻辑
2. 必须判断同类问题是否已经在其他页面、接口、导入数据或文档中同时存在
3. 必须评估本次修改是否会影响：
   - tab 逻辑
   - 页面跳转
   - 云函数
   - 数据结构
   - 导入文件
   - 后台管理端
4. 修复时如果发现同类问题，默认一并修掉，不能只修用户当前指出的那个点
5. 提交前必须重新做一轮最小回归检查，至少覆盖：
   - 当前问题本身
   - 同调用链页面
   - 同数据来源页面
   - 同状态字段或同类文案展示位置

---

## 十二、长期上下文维护要求

为避免终端关闭、对话重开或协作人切换后丢失项目上下文，必须维护项目长期上下文文档。

强制要求：

1. 项目长期上下文统一记录在 `PROJECT_CONTEXT.md`
2. 每次新增功能、重要改动、运行模式变化、遗留问题变化后，必须同步更新 `PROJECT_CONTEXT.md`
3. 后续进入项目时，默认必须先读取：
   - `FINAL_DEVELOPMENT_SPEC.md`
   - `PROJECT_CONTEXT.md`
4. 如果文档内容与临时对话上下文冲突，以最新代码状态和文档为准
5. 不允许只把关键信息留在聊天里而不写入文档

---

## 十三、代码结构与架构清晰度要求

后续所有新增功能、优化功能、重构和修 bug，必须同时满足“代码可读、边界清晰、易维护、可扩展”。

目标标准：

1. 代码仓库必须朝“教科书式结构”靠拢
2. 前端、后台、云函数、数据模型、mock、配置、文档职责必须清晰分层
3. 不允许把多个业务职责长期堆在同一个大文件里
4. 不允许为了快而把临时逻辑、兜底逻辑、页面逻辑、数据逻辑混写成大杂烩
5. 代码应达到让有经验的技术负责人审查时，能快速看懂结构和边界

强制要求：

1. 每个目录必须有明确职责
   - `pages` 只放页面展示和页面交互编排
   - `utils` 只放公共工具、数据访问、通用业务封装
   - `cloudfunctions` 只放云端核心写操作和需要云端保证一致性的逻辑
   - `mock` 只放演示数据和本地调试数据
   - 文档文件只放规范、上下文、说明，不混入业务代码
2. 页面文件必须轻量
   - 页面 `index.js` 优先负责生命周期、页面状态、交互编排
   - 重业务逻辑应抽到独立 service / util
   - 不允许页面文件无限膨胀
3. 云函数必须边界单一
   - 一个云函数只负责一类明确业务动作
   - 参数校验、权限校验、事务、返回结构必须统一
   - 禁止把多种无关业务揉进同一个云函数
4. 数据结构必须统一
   - 同一业务对象字段命名要稳定
   - 前端、云函数、导入数据、后台管理端的字段语义要对齐
   - 若字段变化，必须同时评估上下游影响
5. 变更必须可控
   - 修改某个功能点时，不能悄悄改变其他功能的行为
   - 如果必须改公共逻辑，必须先评估影响范围，再做回归验证
6. 文件必须可维护
   - 函数命名要明确
   - 同类逻辑就近收敛
   - 重复逻辑优先抽取
   - 删除废弃逻辑和过时代码，避免仓库持续变脏
7. 重构不是可选项
   - 如果新增功能会明显加重已有文件混乱度，必须先整理结构再继续写功能
   - 如果发现当前目录或文件职责已经混乱，后续开发时要顺手拆分和收敛

交付要求：

1. 每次新增功能或优化功能后，除了功能可用，还必须说明是否涉及结构优化
2. 每次改动后必须检查是否出现：
   - 页面逻辑侵入公共层
   - 公共层反向依赖页面
   - 云函数职责混乱
   - 前后端字段不一致
   - mock 数据和真实数据结构漂移
3. 如果本次没有做结构整理，也要明确说明当前遗留的结构风险

## 十四、第一阶段正式业务范围

当前项目第一阶段正式业务范围已经明确，后续开发默认按以下范围执行：

1. 玩家浏览首页、主题、活动
2. 玩家进入组局大厅
3. 玩家发起或加入意向组局
4. 门店确认真实到店成员
5. 系统生成真实队伍房间
6. 门店开始场次
7. 门店结束场次
8. 系统自动结算：
   - 档案
   - 徽章
   - 成长值
   - 排行榜
9. 店员上传集锦
10. 玩家回看结果与集锦

当前第一阶段默认不再把以下流程作为主路径：

1. 玩家手动输入打卡码
2. 独立 `admin-web` 作为核心运营入口
3. 重即时聊天能力
4. 支付和订单系统

---

## 十五、门店场次管理强制流程

门店场次管理必须严格按以下五步执行，所有 UI、云函数、状态机和测试默认按此链路设计：

1. `确认成员`
2. `开始场次`
3. `结束场次`
4. `自动结算`
5. `上传集锦`

强制规则：

1. `确认成员` 不是 `开始场次`
2. `开始场次` 和 `结束场次` 必须是两个独立动作
3. `结束场次` 后才能触发自动结算
4. `上传集锦` 必须在结算完成后可见
5. 玩家端不能再把“手动打卡输入”作为主流程入口
6. 任何新页面和按钮文案都不得破坏上述顺序

---

## 十七、第一阶段页面与目录落点要求

第一阶段新增功能默认按业务域拆页，不允许继续把新功能堆进现有单页。

计划页面目录：

1. `pages/lobby/`
2. `pages/lobby-create/`
3. `pages/team-room/`
4. `pages/leaderboard/`
5. `pages/staff-auth-code/`
6. `pages/staff-dashboard/`
7. `pages/staff-session/`
8. `pages/staff-highlights/`

如果某次迭代范围受限，可以合并部分页面，但必须满足：

1. 玩家域和门店域不混在同一个页面文件里
2. 房间域和大厅域不混在同一个页面文件里
3. 场次管理页不能和授权页混在一起

## 十八、UI 与流程一致性要求

第一阶段 UI 必须延续当前小程序已有风格：

1. 暖棕金主色
2. 圆角卡片
3. 档案馆 / 收藏册式视觉语言
4. 胶囊主按钮与浅金次按钮

禁止：

1. 新增另一套冷色后台风
2. 玩家端和门店端视觉完全割裂
3. UI 文案和流程图不一致

所有新页面必须和仓库内以下文档保持一致：

1. `docs/UI_STYLE_GUIDE.md`
2. `docs/UI_FLOW_GUIDE.md`
