# 开发规范

## 一、默认优先级

所有后续开发，默认按以下优先级决策：

1. 安全性
2. 性能
3. 用户体验

## 二、异常场景防御

以下异常场景必须全部实现：

1. 弱网/断网
   - 必须有自动提示
   - 必须提供重试按钮
   - 必须有无网/请求失败兜底文案

2. 并发冲突
   - 必须防重复提交
   - 必须加锁或幂等控制
   - 必须保证数据一致性

3. 误操作
   - 删除/取消/清空类操作必须二次确认

4. 输入校验
   - 必须校验空值
   - 必须限制超长文本
   - 必须过滤特殊字符或非法输入

5. 云开发异常
   - 数据库失败必须捕获
   - 云函数超时必须捕获
   - 必须给用户友好提示

6. 无权限/未登录
   - 必须自动引导
   - 页面不能崩溃
   - 不能直接抛原始错误给用户

## 三、AI 开发强制约束

以下规则绝对禁止违反：

1. 禁止只写逻辑不写代码
2. 禁止甩大段无注释代码
3. 所有异步请求必须包 `try/catch`
4. 所有接口必须加加载提示
5. 所有错误必须给用户友好提示
6. 必须考虑测试视角：边界、异常、空值、重复点击
7. 代码风格统一：ES6+、`async/await`、严格模式
8. 云开发必须正确配置 `envId`，禁止硬写危险代码
9. 页面必须做：`onLoad` 异常捕获、无数据处理、网络异常处理
10. 每次输出必须包含：

- 功能说明
- 代码
- 测试点
- 验收标准

## 四、前端页面最低要求

每个页面默认至少要考虑：

- `onLoad` 使用异常捕获
- 空状态展示
- 加载中状态
- 网络异常提示
- 重试入口
- 关键操作防重复点击

## 五、云函数最低要求

所有云函数默认必须满足：

1. 必须包含事务处理
2. 多文档更新必须放在事务中
3. 必须做参数校验
4. 必须做身份校验和权限校验
5. 必须考虑幂等
6. 必须记录可排查日志
7. 必须返回统一成功/失败结构

建议默认结构：

1. 参数校验
2. 身份校验
3. 权限校验
4. 开启事务
5. 查询当前状态
6. 执行业务更新
7. 提交事务
8. 返回结果

## 六、测试验收规范

**每次修改代码后，必须完成以下测试才能说"通过"：**

### 1. 本地 Mock 测试（必须）

```bash
npm run test:regression
```

- 验证代码逻辑正确性
- 验证数据结构完整性
- 验证边界场景处理

### 2. 云端数据测试（必须）

- 在微信开发者工具中，将 `app.js` 切换为：
  ```javascript
  useMockData: false,
  useMockGroups: false,
  ```
- 部署相关云函数（如果有修改）
- 在真机上测试核心业务流程：
  - 创建组局 → 取消 → 删除
  - 加入组局 → 退出 → 删除
  - 门店授权 → 场次管理
  - 个人档案 → 排行榜

### 3. 通过标准

- ✅ 本地 mock 测试全部通过
- ✅ 云端数据测试核心流程正常
- ✅ 无 console.error 报错
- ✅ 无用户可见的异常行为

**只有两项测试都通过，才能说"修复完成"。**

## 七、Mock 数据与云端数据隔离规范（强制）

**背景**：2026-03-29 修复了一例因 mock 数据泄漏导致的徽章数量不一致 bug。mock 数据中的 `playRecords: []` 在云端模式下混入真实数据，导致客户端徽章计算永远按空数据执行。为避免同类问题反复出现，制定以下隔离规范。

### 核心原则

**Mock 数据只在本地测试时使用，绝对不能泄漏到线上云端模式。**

### 具体规则

1. **禁止用 mock 数据做默认值兜底**
   - 客户端 `normalizeProfile`、`normalizeGroupItem` 等函数中，禁止使用 `cloneDefaultProfile()` 或引用 `mockData.xxx` 作为合并基底
   - 云端模式下应以云端返回的数据为准，不需要合并 mock 默认值

2. **云端云函数必须维护独立的 buildDefault**
   - 每个云函数（`getProfile`、`getLeaderboard`、`staffManage` 等）必须有自己的 `buildDefaultProfile()` 函数
   - 禁止云函数引用小程序端的 `mock/data.js`

3. **字段名统一**
   - 云端存储统一使用 `punchRecords` 作为打卡记录字段名
   - 禁止在云端数据中出现 `playRecords`，该字段仅限 mock 数据使用
   - 客户端读取时优先检查 `punchRecords`，fallback 到 `playRecords` 仅用于兼容旧 mock 数据

4. **模式判断前置**
   - 所有涉及数据读取的函数，必须先判断 `useMock()` 再决定数据源
   - 云端模式下不加载、不引用、不依赖任何 mock 文件

5. **新增字段必须同步**
   - 云端 `buildDefaultProfile` 新增字段时，mock 数据和客户端默认值也需同步
   - 云端 `buildSessionRecord` 新增字段时，客户端 `buildProfileStats` 的读取逻辑也需同步

### 审查清单

新增或修改涉及 profile / group / session 数据的代码时，必须检查：

- [ ] 是否有地方引用了 `mockData` 或 `cloneDefaultProfile()`
- [ ] 该引用是否在 `useMock() === false` 时也会执行
- [ ] 云端和客户端的字段名是否一致
- [ ] 默认值合并是否会覆盖真实数据
