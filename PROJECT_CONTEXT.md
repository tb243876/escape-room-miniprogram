# 项目长期上下文

本文件沉淀项目长期业务口径、用户要求和架构边界。

## 1. 项目范围

- 小程序目录：`/Users/edy/Documents/密室小程序/miniapp`
- Web 后台目录：`/Users/edy/Documents/密室小程序/admin-web`
- 当前第一阶段主战场是 `miniapp`
- `admin-web` 仅保留历史原型参考，不再承担当前主链路开发

## 2. 当前业务目标

第一阶段目标是做一个统一小程序多角色方案：

1. 玩家浏览主题和活动
2. 玩家发起或加入组局
3. 门店在工作台确认真实到店成员
4. 门店开始和结束场次
5. 系统自动结算档案、成长值、徽章、排行榜
6. 门店上传集锦，玩家在有效期内查看

明确下线的旧方向：

- 玩家手动输入打卡码
- `admin-web` 作为核心运营入口

## 3. 当前运行策略

当前真实运行口径：

- `release -> prod`
- `trial -> prod`
- `develop -> prod`

当前关键配置：

- `useMockData: false`
- `useMockGroups: false`
- 所有运行版本禁止 mock fallback
- `test/_test` 已退出当前项目运行口径，体验版、本地开发、正式版统一只读写 `trial/prod` 真数据集合

当前运营目标：

- 体验版必须可以直接拿到线下使用
- 不依赖测试种子来证明业务闭环

## 4. 用户长期要求

这是本项目的高优先级长期要求：

1. 每次改问题前必须先审查相关链路，不能只修单点
2. 修一个问题时要同步检查同类问题会不会出现在其他页面或逻辑里
3. 不要用临时假方案糊弄界面或交互
4. 页面体验优先考虑真实可用，不要做“看起来解决了”的绕路方案
5. 详情页可以是完整页面，不要强行做成半弹层
6. 布局要自适应，不能靠硬顶高度或大块留白凑效果
7. 每次新增功能、重要改动、运行模式变化、已知遗留问题，都要同步更新本文件
8. 每次新增功能或优化功能时，必须同时整理代码结构，保证前端、后台、云函数、mock、配置、文档边界清晰
9. 仓库要持续向“教科书式、可维护、职责分明”收敛
10. 修改一个功能点时，必须评估是否会影响其他功能点
11. 视觉设计必须延续当前暖棕金、圆角卡片、档案馆式风格
12. 每次改动都必须记录 `CHANGELOG.md`

## 6. 成长体系规则

当前成长体系的真实口径来自 `utils/domain/profile.js`、`cloudfunctions/getProfile/profile-domain.js`、`cloudfunctions/staffManage/profile-domain.js`。

### 6.1 等级

按累计真实场次 `totalPlayCount` 计算：

| 场次区间 | 等级 |
| --- | --- |
| `0-2` | 新客玩家 |
| `3-5` | 沉浸玩家 |
| `6-9` | 进阶会员 |
| `10+` | 馆藏玩家 |

### 6.2 成长值

- 当前每次正式结算默认 `+18 growthValue`
- 入口是 `staffManage -> runSessionAction('end') -> applySettlement`
- 结算是幂等的，同一 `sessionId` 不应重复加分

### 6.3 连续活跃

- `streakDays` 按 `lastPlayedAt` 与本次结算日期差计算
- 同一天重复结算不增加
- 间隔 1 天则续增
- 间隔超过 1 天则重置为 1

### 6.4 徽章

当前已落地徽章规则包含以下几类：

- 场次数量：
  - `1 / 3 / 5 / 10` 场
- 指定主题：
  - `瞳灵人`
  - `文川中学`
- 恐怖等级：
  - 微恐 / 中恐 / 重恐 / 全级别通关
- 时间与组队：
  - 深夜场
  - 4 人及以上组队
  - 双人场
- 主题覆盖：
  - 3 个不同主题
  - 6 个不同主题
- 时间窗口：
  - 30 天 3 场
  - 365 天 6 场

### 6.5 荣誉与称号

- `titleLabel` 默认由等级和高价值徽章推导
- `honorLabels` 默认取高价值徽章、组队活跃、近 30 天活跃等信号
- 如果后续要改展示文案，必须同时改前端和云端规则，不允许一边显示一边结算各算各的

## 8. 多门店支持口径

当前版本明确是单店口径：

- 默认 `storeName = 迷场档案馆`
- 玩家主键仍然只用 `openid`
- 当前集合结构没有强制 `storeId`

这意味着：

- 当前不要假装已经支持跨店隔离
- 当前所有查询、权限、索引默认按单店设计

如果后续要升级多门店，必须一次性补齐：

1. `groups / staff_sessions / staff_highlights / staff_bindings / staff_auth_codes` 全量补 `storeId`
2. 所有工作台查询默认带 `storeId`
3. 所有复合索引把 `storeId` 放到前缀
4. 权限矩阵改成“角色 + 门店范围”双重约束

## 9. 当前高风险关注点

后续任何改动前都要优先核对：

1. 组局、房间、场次是否仍按 `STATE_MACHINE.md` 流转
2. 资料卡、档案页、门店用户看板是否仍然展示同一套资料口径
3. 体验版是否仍然走 `prod`，且没有 mock fallback
4. `groups.participants` 是否仍然是成员真相源
5. 删除记录是否仍然只写 `hiddenForOpenIds`
